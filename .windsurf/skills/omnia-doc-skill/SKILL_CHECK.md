## 2. How the AI Should Use This Document

The AI must reference this document during the **CHECK** phase (before finalizing):

**During CHECK (before finalizing):**
- Run the self-review against §3 (Guardrails and Constraints) checklists
- Flag uncertain content with RST review comments (see §3.4)
- Verify file placement against the Information Map (from COLLECT phase)
- Apply confidence thresholds (§3.3) to determine review needs
- Use §7 (Check Process Workflow) for systematic validation
- **Verify content update vs. creation decision** — Ensure that existing content was updated rather than creating duplicate new content

For all writing conventions, voice, tone, and formatting standards not explicitly covered in this skill document, refer to the Omnia Style Guide:

## 3. Guardrails and Constraints

### 3.1 Always Do

- [ ] Include a `Prerequisites` section in every How-To/Procedure topic (NOT for Concept topics)
- [ ] Specify the language in every `.. code-block::` directive
- [ ] Add the new file to the parent `index.rst` toctree
- [ ] Add a unique RST label to the top of every file: `.. _[unique-label]:`
- [ ] Use only approved terminology from BUILD phase §6
- [ ] Include a `Related Topics` section at the bottom of every page
- [ ] Use exact configuration parameter names with double backticks
- [ ] Add `.. versionadded::` or `.. versionchanged::` when content is version-specific
- [ ] Validate placement against the Information Map (from COLLECT phase) before creating a new file
- [ ] Write from source assets (Behaviour Spec → Engineering Specification → Component Specification → HLD → Engineering Notes → Demo Transcripts → Unit Tests) — never from general knowledge alone
- [ ] Verify that existing content was updated rather than creating duplicate new content (check for related topics before creating new files)

### 3.2 Never Do

- [ ] Use bare URLs to link between internal documentation pages
- [ ] Include prerequisites section on Concept topics (Concept topics should NOT have prerequisites)
- [ ] Reference internal team names, personal names, or internal system aliases
- [ ] Include internal repo paths not intended for end users
- [ ] Use marketing superlatives: "powerful", "seamless", "industry-leading", "effortlessly"
- [ ] Combine multiple actions into a single numbered step
- [ ] Create an `.rst` file without registering it in a toctree
- [ ] Use `.. note::` as a container for main content
- [ ] Invent parameter names — always verify against Behaviour Spec, Engineering Specification, or actual config files
- [ ] Write troubleshooting content that blames the user

### 3.3 Confidence Thresholds

| Confidence Level | Threshold | AI Action                                                          |
|------------------|-----------|--------------------------------------------------------------------|
| High             | > 0.85    | Generate normally                                                  |
| Medium           | 0.6–0.85  | Generate with `.. AI_REVIEW::` comment markers                    |
| Low              | < 0.6     | Flag entire section for human authoring; provide outline only      |

### 3.4 Review Markers

When uncertain, insert an RST comment:

```rst
.. AI_REVIEW: [reason for uncertainty]
```

**Common reasons:**
- `Inferred from Behaviour Spec only — verify against Engineering Specification or Component Specification`
- `No Behaviour Spec found for this component — SME verification required`
- `Parameter name not confirmed — check Behaviour Spec, Engineering Specification, or actual config files`
- `Multiple possible workflows — confirm correct path with SME`
- `Placement uncertain — verify toctree location`
- `Version applicability unclear — confirm which Omnia version`

## 6. Quick Reference Cards

### For SMEs and Technical Reviewers

When reviewing AI-generated content, check:
- Terminology matches BUILD phase §6 (Terminology and Glossary)
- Technical accuracy of all commands and parameter names
- Gotchas and edge cases not covered — add to §4
- Deprecated features that need documentation — add to §4

### For WindSurf SWE (AI Content Generation)

**Context assembly order before generating:**

1. Load this skill document (`docs/.windsurf/skills/omnia-doc-skill/SKILL_CHECK.md`)
2. Load the Omnia Style Guide (`review_source/dell-style-guide.md`) for general writing standards
3. Load the relevant source assets from `knowledge_source/victoria_logs_cluster_mode/` (Behaviour Spec → Engineering Specification → Component Specification → HLD → Engineering Notes → Demo Transcripts → Unit Tests)
4. Load existing `.rst` files in the target section to match style and avoid duplication
5. Identify content type using BUILD phase §5.3 and file placement using COLLECT phase §4
6. Generate draft following RST conventions in BUILD phase §5
7. Self-review against §3 (Guardrails and Constraints) checklists
8. Insert `.. AI_REVIEW:` markers for uncertain content
9. Output with recommended file path and toctree placement

**Regular maintenance schedule:**

| Cadence     | Task                                                                 |

## 7. Check Process Workflow

### 7.1 Pre-Validation Checklist

Before marking content as complete, run through this checklist:

**Content Validation:**
- [ ] All technical facts verified against source assets (Behaviour Spec > Engineering Specification > Component Specification > HLD > Engineering Notes > Demo Transcripts > Unit Tests)
- [ ] Terminology matches approved glossary (BUILD phase §6)
- [ ] Commands and parameters are exact and tested
- [ ] Version-specific content properly marked with `.. versionadded::` or `.. versionchanged::`

**Structure Validation:**
- [ ] File follows RST structure template (BUILD phase §5.1)
- [ ] Heading hierarchy is correct (BUILD phase §5.2)
- [ ] All code blocks specify language
- [ ] Cross-references use `:doc:` or `:ref:` directives
- [ ] Unique RST label present at top of file

**Placement Validation:**
- [ ] Content type matches placement decision (COLLECT phase §4)
- [ ] File name follows naming conventions (COLLECT phase §4)
- [ ] File registered in appropriate toctree
- [ ] Parent index.rst updated if new section

**Quality Validation:**
- [ ] Prerequisites section present and complete (for How-To/Procedure topics only - NOT for Concept topics)
- [ ] Verification step present in all How-To/Procedure topics
- [ ] Related Topics section present with valid links
- [ ] No marketing language or superlatives
- [ ] Voice and tone appropriate for target audience
- [ ] No internal team names or aliases
- [ ] All admonitions (warning, note, tip) added where appropriate
- [ ] Flagging system used correctly: [TO BE PROVIDED] and [SME VALIDATION REQUIRED]

### 7.2 Confidence Assessment

After completing the pre-validation checklist, assess confidence:

**High Confidence (> 0.85):**
- All checklist items complete
- Source assets clear and comprehensive
- No technical ambiguities
- Examples verified against working system

**Medium Confidence (0.6–0.85):**
- Minor gaps in source assets
- Some technical ambiguities
- Need SME verification on specific points
- Insert `.. AI_REVIEW:` markers for uncertain areas

**Low Confidence (< 0.6):**
- Major gaps in source assets
- Significant technical ambiguities
- Missing critical information
- Provide outline only, flag for human authoring

### 7.3 Review Marker Guidelines

Use `.. AI_REVIEW:` markers strategically:

**When to use:**
- Parameter names not confirmed in HLD or Engineering Notes
- Workflow steps inferred from limited sources
- Version applicability unclear
- Multiple possible approaches documented
- Technical assumptions made without verification

**When NOT to use:**
- Standard formatting or style questions
- Minor wording improvements
- General clarification needs
- Non-technical content decisions

**Review marker format:**
```rst
.. AI_REVIEW: Parameter 'network_timeout' value inferred from demo transcript - verify against Engineering Notes
```

### 7.4 Final Output Requirements

Every completed piece of content must include:

1. **Complete RST content** ready for publication
2. **Recommended file path** relative to `docs/`
3. **Toctree entry** for parent `index.rst`
4. **AI_REVIEW markers list** with reasons (if any)
5. **Confidence assessment** and rationale

**Example output format:**
```
=== CONTENT ===
[Complete RST content here]

=== METADATA ===
File path: docs/how-to/configuration/configure-scheduler.rst
Toctree entry: how-to/configuration/configure-scheduler
Confidence: High (0.9)
AI_REVIEW markers: None
```
