## 3. Product Identity

### What Omnia Does

Omnia is a containerized, open-source deployment toolkit designed to automate the setup and management of high-performance computing (HPC) environments on Linux-based servers. It leverages Ansible playbooks to streamline:

* Operating system provisioning
* Driver installation and configuration
* Deployment of workload schedulers such as Slurm and Kubernetes
* Installation of optimization libraries, machine learning frameworks, and AI models
* Management of compute, storage, and networking resources

Omnia simplifies infrastructure deployment in complex environments, enabling faster setup and consistent configuration across systems. It is designed for on-premise HPC deployments and is typically used in data centers running large-scale compute clusters.

### The One-Sentence Summary

> **Omnia simplifies infrastructure deployment in complex HPC environments, enabling faster setup and consistent configuration across systems.**

### Key Capabilities

For a complete list of capabilities, see: [New Features — Dell/Omnia](https://omnia-docs.readthedocs.io/)

|------------------------|------------------------------------------|------------------------|
| OS Provisioning        | Automated operating system installation  | Concept / How-To       |
| Scheduler Deployment   | Slurm and Kubernetes setup               | How-To / Reference     |
| Driver Management      | Automated driver installation            | How-To                 |
| ML/AI Framework Setup  | ML libraries and AI model deployment     | How-To / Reference     |
| Resource Management    | Compute, storage, network orchestration  | Concept / How-To       |

### Market Position

Omnia is an open-source toolkit specifically designed for HPC infrastructure automation. It differentiates itself by providing a comprehensive, Ansible-based solution that handles the full stack from bare-metal provisioning to AI framework deployment.

## 5. RST Documentation Conventions

### 5.1 File Structure Template

Every `.rst` file must follow this structure:

```rst
.. _[unique-label]:

[Page Title in Title Case]
==========================

Brief one- or two-sentence introduction stating what this page covers
and who it is for.

.. contents:: On This Page
   :local:
   :depth: 2

Prerequisites
-------------

Before you begin, ensure:

* [Prerequisite 1]
* [Prerequisite 2]

[Section Heading]
-----------------

Content here.

.. note::

   Use notes sparingly for genuinely important information.

Related Topics
--------------

* :doc:`[related-page]`
* :doc:`[another-page]`
```

### 5.2 Heading Hierarchy

RST headings are defined by underline characters. Use this order consistently:

```
Page Title        →  ========  (equals sign, full width)
Section           →  --------  (dash, full width)
Subsection        →  ~~~~~~~~  (tilde, full width)
Sub-subsection    →  ^^^^^^^^  (caret, full width)
```

**Rule:** Never skip a heading level. Never use more than four levels on a single page.

### 5.3 Topic Type Guidelines

**Concept Topics:**
- Purpose: Explain what something is, why it matters, and how it works
- Structure: Definition → Why It Matters → How It Works
- **No prerequisites section** - focus purely on conceptual understanding
- Use diagrams and examples to clarify complex concepts
- Link to related how-to topics for practical application

**Procedure (How-To) Topics:**
- Purpose: Guide users through completing a specific task
- Structure: Prerequisites → Steps → Result → Next Steps
- **Always include verification step** to confirm task completion
- **Source priority for workflow steps:** Engineering Notes → Unit Tests → Demo Transcriptions → HLD
- Use numbered steps with specific commands and configurations
- Include expected outcomes and troubleshooting

**Reference Topics:**
- Purpose: Provide detailed technical reference information
- Structure: Overview → Parameters/Commands → Examples → Related Topics
- Include comprehensive parameter tables and command syntax
- Use code blocks with language specification

**Troubleshooting Topics:**
- Purpose: Help users resolve common issues
- Structure: Symptoms → Causes → Solutions → Prevention
- Include error messages and log excerpts
- Provide step-by-step resolution procedures

### 5.4 Step Writing Rules (How-To Topics)

- Use **numbered lists** (`#.`) for ordered steps — never bullets for procedural content
- Each step = **one action**. Split compound actions into separate steps
- Begin each step with an **imperative verb**: "Run", "Edit", "Set", "Copy", "Navigate"
- Include the **expected result** after steps where verification is possible
- Use `.. code-block::` for all commands — never inline backtick-only code in steps
- **Always include a verification step** after the main procedure to confirm task completion
- **Source information priority:** Extract workflow steps from Engineering Notes first, then Unit Tests, then Demo Transcriptions, then supplement with HLD if needed
- **Missing information handling:** Use `<To be decided>` for any steps, prerequisites, or verification information that cannot be found in source materials

```rst
#. Install the required packages.

   .. code-block:: bash

      sudo dnf install -y omnia-control-plane

#. Verify the installation completed successfully.

   .. code-block:: bash

      omnia --version

   The output should display the installed version, for example: ``omnia 4.2.0``.
```

### 5.5 Code Blocks

Always specify the language:

```rst
.. code-block:: bash        ← Shell commands
.. code-block:: yaml        ← Configuration files
.. code-block:: json        ← JSON payloads
.. code-block:: python      ← Python scripts
.. code-block:: ini         ← INI/conf files
.. code-block:: text        ← Output / logs (no syntax highlighting)
.. code-block:: console     ← Interactive terminal sessions with prompts
```

**Rules:**
- Never use bare backtick blocks for multi-line commands
- Use `.. literalinclude::` to embed actual files from the repo (preferred for config examples)
- Use `.. code-block:: text` for sample output, never `bash`

### 5.6 Admonitions

Use sparingly. Each admonition must earn its place.

| Admonition       | Use For                                             | Frequency     |
|------------------|-----------------------------------------------------|---------------|
| `.. note::`      | Helpful context that is not critical                | ≤ 1 per page  |
| `.. warning::`   | Actions that could cause data loss or system issues | Only when true|
| `.. important::` | Prerequisites or dependencies that are easily missed| ≤ 1 per page  |
| `.. tip::`       | Optional shortcuts or best practices                | ≤ 1 per page  |
| `.. caution::`   | Potential for irreversible changes                  | Only when true|

**Never use `.. note::` as a substitute for main content.** If the information is important enough to note, it belongs in the body.

### 5.7 Cross-References

Use RST labels and `:doc:` or `:ref:` directives — never bare URLs to internal pages.

```rst
See :doc:`../reference/cli/omnia-provision` for CLI options.

For background, see :ref:`concept-cluster-topology`.
```

**Label naming convention:** `[topic-type]-[noun-phrase]`
- `concept-cluster-topology`
- `task-configure-network`
- `ref-omnia-cli`

### 5.8 Voice and Tone Rules

**General Principle:** Follow the Omnia Style Guide for general writing standards. The rules below are Omnia documentation-specific applications and clarifications.

**Style Guide Reference:** See `docs/STYLE_GUIDE/Dell Technologies Unified Style Guide v1.0.pdf` for comprehensive voice and tone guidance.

| Rule                          | Correct                                      | Incorrect                                   |
|-------------------------------|----------------------------------------------|---------------------------------------------|
| Use second person             | "Run the following command"                  | "The user should run..."                    |
| Use imperative mood for steps | "Configure the inventory file"               | "You can configure the inventory file"      |
| Use active voice              | "Omnia provisions the nodes"                 | "The nodes are provisioned by Omnia"        |
| Be direct and specific        | "Set `scheduler_type` to `slurm`"            | "Configure the scheduler appropriately"     |
| No marketing language         | "Omnia supports 200+ nodes per cluster"      | "Omnia's powerful engine scales effortlessly"|
| Neutral tone in troubleshooting | "This error occurs when..."               | "Unfortunately, this known issue..."        |
| Lead with the action          | "To configure X, edit the `omnia.yml` file"  | "The `omnia.yml` file can be used to configure X" |

### 5.9 ReadTheDocs / Sphinx-Specific Rules

- Use `conf.py` extensions: `sphinx.ext.intersphinx`, `sphinx_rtd_theme`, `myst_parser` (if Markdown mixed)
- Include `:orphan:` at the top of any `.rst` file intentionally excluded from a toctree
- Use `.. versionadded:: X.Y` and `.. versionchanged:: X.Y` for version-specific content
- Use `.. deprecated:: X.Y` for deprecated features, with a migration note
- All images must use `.. figure::` with an `:alt:` attribute and a caption

### 7.1 Good RST Examples

#### Concept Topic — Exemplary

```rst
.. _concept-cluster-provisioning:

Cluster Provisioning
====================

Cluster provisioning is the process by which Omnia configures bare-metal
nodes — installing operating systems, network settings, and required
software — to function as part of a managed cluster.

.. contents:: On This Page
   :local:
   :depth: 2

How Provisioning Works

* **Star topology:** All compute nodes connect directly to the control plane
  for centralized management and monitoring
* **Mesh topology:** Compute nodes interconnect for peer-to-peer communication
  and workload distribution
* **Hybrid topology:** Combines star and mesh elements for balanced
  management and performance

The topology is defined in the ``omnia.yml`` configuration file using the
``cluster_topology`` parameter. Each topology type has specific network
requirements and performance characteristics.

**Related topics:**
* :doc:`../how-to/configuration/configure-topology`
* :doc:`../reference/topology-parameters`
```

**Why it's good:** Defines the concept clearly, explains the mechanism step-by-step, uses approved terminology, links to related topics, and **does not include prerequisites** (focuses purely on conceptual understanding).

#### How-To Topic — Exemplary

```rst
.. _task-configure-scheduler:

Configuring the Job Scheduler
==============================

Configure the job scheduler to enable workload management across the
cluster. Omnia supports Slurm and PBS Professional.

.. contents:: On This Page
   :local:
   :depth: 2

Prerequisites
-------------

Before configuring the scheduler:

* The control plane must be installed and running (see :doc:`../getting-started/installation`)
* All compute nodes must be provisioned (see :doc:`provision-nodes`)
* You must have root or sudo access on the control plane

Procedure
---------

#. Navigate to the Omnia configuration directory.

   .. code-block:: bash

      cd /opt/omnia/config

#. Open the scheduler configuration file.

   .. code-block:: bash

      sudo vi omnia.yml

#. Set the ``scheduler_type`` parameter to your preferred scheduler.

   .. code-block:: yaml

      scheduler_type: slurm

#. Save the configuration file and exit the editor.

Result
------

The scheduler type is now configured in the Omnia configuration file.
The next time Omnia services restart, they will use the specified scheduler
for workload management.

Verification
------------

Verify the scheduler configuration is valid:

.. code-block:: bash

   omnia-cli validate --check scheduler

Expected output:

.. code-block:: text

   ✓ Scheduler configuration is valid
   ✓ Slurm integration ready

If the validation fails, check the ``omnia.yml`` file syntax and ensure
the specified scheduler type is supported.

Next Steps
-----------

* Restart Omnia services to apply the new scheduler configuration
* Configure scheduler-specific settings (queues, partitions, etc.)
* Submit test jobs to verify scheduler operation

**Related topics:**
* :doc:`../reference/scheduler-parameters`
* :doc:`../troubleshooting/scheduler-issues`)
```

**Why it's good:** Includes clear prerequisites, step-by-step procedure, **verification step** to confirm completion, expected results, and next steps. **Source priority:** Workflow steps extracted from Engineering Notes → Unit Tests → Demo Transcriptions → HLD. Uses `<To be decided>` for any missing information.

#### Reference Topic — Exemplary

```rst
.. _ref-omnia-provision:

omnia provision
===============

The ``omnia provision`` command configures bare-metal nodes by installing
the operating system, network settings, and required software components
to function as part of a managed Omnia cluster.

.. contents:: On This Page
   :local:
   :depth: 2

Synopsis
--------

.. code-block:: bash

   omnia provision [OPTIONS] NODE_SPEC

Description
-----------

The ``omnia provision`` command configures bare-metal nodes by installing
the operating system, network settings, and required software components
to function as part of a managed Omnia cluster.

Options
-------

.. list-table::
   :header-rows: 1
   :widths: 25 15 60

   * - Option
     - Type
     - Description
   * - ``--inventory``
     - string
     - Path to the Ansible inventory file. Default: ``/opt/omnia/inventory``
   * - ``--dry-run``
     - flag
     - Simulate provisioning without making changes.
   * - ``--force``
     - flag
     - Re-provision nodes that are already configured.
   * - ``--log-level``
     - string
     - Verbosity level: ``info``, ``debug``, ``warn``. Default: ``info``

Examples
--------

Provision all nodes in the inventory:

.. code-block:: bash

   omnia provision --inventory /opt/omnia/inventory

Provision specific nodes:

.. code-block:: bash

   omnia provision node01 node02 node03

Simulate provisioning without changes:

.. code-block:: bash

   omnia provision --dry-run

Related Topics
--------------

* :doc:`../how-to/configuration/provision-nodes`
* :doc:`../concepts/cluster-provisioning`
```

**Why it's good:** Consistent structure, exact option names, default values documented, real examples.

### 7.2 Anti-Patterns to Avoid

| Anti-Pattern              | Bad Example                                                | Problem                                      | Correct Approach                                   |
|---------------------------|-----------------------------------------------------------|----------------------------------------------|----------------------------------------------------|
| Vague steps               | "Configure the network settings appropriately"            | Not actionable — user doesn't know what to do| "Set ``network_interface`` to ``ens3f0`` in `omnia.yml`"|
| Missing prerequisites     | (Steps with no prerequisites section)                     | User encounters unexpected failures          | List all required access, installed software, and prior steps |
| Marketing language        | "Omnia's powerful, industry-leading provisioning engine"  | Not factual, not useful                      | "Omnia provisions nodes using Ansible playbooks"   |
| Passive voice             | "The scheduler should be configured before nodes are added"| Unclear who acts and when                  | "Configure the scheduler before adding nodes"      |
| Bare URLs                 | `See https://docs.omnia.io/reference/cli`                 | Breaks on restructure; not semantic          | `See :doc:`../reference/cli/omnia-cli``            |
| Undeclared acronyms       | "Use the BMC to configure IPMI over the OOB VLAN"        | Inaccessible to new users                    | Spell out on first use per document                |
| Generic code blocks       | `` ` `` without language                                  | No syntax highlighting; unclear context      | Always use `.. code-block:: bash` (or appropriate language) |
| Step combining            | "Open the file and change the scheduler setting and save" | Skips verification opportunities             | One action per numbered step                       |
