## 3. Product Identity

### What Omnia Does

Omnia is a containerized, open-source deployment toolkit designed to automate the setup and management of high-performance computing (HPC) environments on Linux-based servers. It leverages Ansible playbooks to streamline:

* Operating system provisioning
* Driver installation and configuration
* Deployment of workload schedulers such as Slurm and Kubernetes
* Installation of optimization libraries, machine learning frameworks, and AI models
* Management of compute, storage, and networking resources

Omnia simplifies infrastructure deployment in complex environments, enabling faster setup and consistent configuration across systems. It is designed for on-premise HPC deployments and is typically used in data centers running large-scale compute clusters.

### The One-Sentence Summary

> **Omnia simplifies infrastructure deployment in complex HPC environments, enabling faster setup and consistent configuration across systems.**

### Key Capabilities

For a complete list of capabilities, see: [New Features — Dell/Omnia](https://omnia-docs.readthedocs.io/)

| Capability             | Description                              | Primary Doc Type       |
|------------------------|------------------------------------------|------------------------|
| OS Provisioning        | Automated operating system installation  | Concept / How-To       |
| Scheduler Deployment   | Slurm and Kubernetes setup               | How-To / Reference     |
| Driver Management      | Automated driver installation            | How-To                 |
| ML/AI Framework Setup  | ML libraries and AI model deployment     | How-To / Reference     |
| Resource Management    | Compute, storage, network orchestration  | Concept / How-To       |

### Market Position

Omnia is an open-source toolkit specifically designed for HPC infrastructure automation. It differentiates itself by providing a comprehensive, Ansible-based solution that handles the full stack from bare-metal provisioning to AI framework deployment.

## 4. Target Audiences

### Primary Audience: Infrastructure / HPC Administrator

| Attribute             | Description                                                                 |
|-----------------------|-----------------------------------------------------------------------------|
| Technical Level       | Advanced                                                                    |
| Typical Job Titles    | HPC Administrator, Infrastructure Engineer, Systems Administrator, DevOps Engineer |
| Primary Goals         | Deploy and configure Omnia clusters reliably; automate repetitive setup tasks |
| Pain Points           | Manual provisioning is error-prone and slow; unclear dependency order; lack of rollback guidance |
| Documentation Needs   | Step-by-step How-To guides, prerequisite checklists, configuration reference, troubleshooting |

**Voice/Tone for this audience:**
Write as a knowledgeable peer. Assume Linux proficiency, familiarity with networking concepts, and comfort with YAML/JSON. Do not over-explain basic sysadmin concepts. Be direct, precise, and task-focused. Use imperative mood for instructions.

**Example user story:**
> "As an HPC administrator, I want clear, step-by-step instructions on how to deploy additional container images and packages on Service Kubernetes Cluster and Slurm Nodes."

## 6. Information Map

> WindSurf has direct access to the repository structure and can read the existing file tree. This section defines **placement intent, decision rules, and naming conventions** only — not directory paths.

### Content Type Decision Rules

When deciding where new content belongs, answer these questions in order:

| Question                                      | Answer → Section          |
|-----------------------------------------------|---------------------------|
| Does it explain "what is X" or "why X exists"?| `concepts/`               |
| Is it a first-time setup or installation step? | `getting-started/`        |
| Does it walk through "how do I do X"?          | `how-to/[area]/`          |
| Is it a lookup table (commands, params, API)?  | `reference/`              |
| Does it describe a known problem and fix?      | `troubleshooting/`        |
| Is it a high-level product or design overview? | `overview/`               |

**When content spans two types** (e.g., a concept that also has steps): create two separate files — one concept, one how-to — and cross-reference them.

### File Naming Conventions

| Section          | Naming Pattern                          | Examples                                      |
|------------------|-----------------------------------------|-----------------------------------------------|
| `concepts/`      | Noun phrase, kebab-case                 | `cluster-provisioning.rst`, `job-scheduling.rst` |
| `getting-started/`| Step or milestone name                 | `prerequisites.rst`, `quick-start.rst`        |
| `how-to/`        | Gerund + noun, kebab-case              | `configuring-network.rst`, `adding-nodes.rst` |
| `reference/cli/` | Exact command name                      | `omnia-provision.rst`, `omnia-status.rst`     |
| `reference/api/` | Resource name                           | `nodes.rst`, `clusters.rst`                   |
| `reference/config/`| Config file name                      | `omnia-yml.rst`, `inventory.rst`              |
| `troubleshooting/`| Symptom or error, kebab-case           | `nodes-not-provisioning.rst`                  |

### Toctree Rules

Every new `.rst` file **must** be registered in the `toctree` of its parent `index.rst`. WindSurf must always update the parent `index.rst` alongside any new file it creates. Also update the top-level `index.rst` if an entirely new section is added.

**Example toctree entry:**

```rst
.. toctree::
   :maxdepth: 2
   :caption: How-To Guides

   how-to/configuration/configure-network
   how-to/configuration/configure-storage
```

## 7. RST Documentation Conventions

### 7.1 File Structure Template

Every `.rst` file must follow this structure:

```rst
.. _[unique-label]:

[Page Title in Title Case]
==========================

Brief one- or two-sentence introduction stating what this page covers
and who it is for.

.. contents:: On This Page
   :local:
   :depth: 2

Prerequisites
-------------

Before you begin, ensure:

* [Prerequisite 1]
* [Prerequisite 2]

[Section Heading]
-----------------

Content here.

.. note::

   Use notes sparingly for genuinely important information.

Related Topics
--------------

* :doc:`[related-page]`
* :doc:`[another-page]`
```

### 7.2 Heading Hierarchy

RST headings are defined by underline characters. Use this order consistently:

```
Page Title        →  ========  (equals sign, full width)
Section           →  --------  (dash, full width)
Subsection        →  ~~~~~~~~  (tilde, full width)
Sub-subsection    →  ^^^^^^^^  (caret, full width)
```

**Rule:** Never skip a heading level. Never use more than four levels on a single page.

### 7.3 Topic Types and When to Use Them

| Topic Type    | RST Structure                              | Use When                                              | Title Convention                     |
|---------------|--------------------------------------------|-------------------------------------------------------|--------------------------------------|
| Overview      | Intro → Context → Key Points → Next Steps  | Explaining what something is at a high level          | "Understanding [X]" or "[X] Overview"|
| Concept       | Definition → Why It Matters → How It Works | Explaining a mechanism or design decision             | "[Noun phrase]" or "About [X]"       |
| How-To        | Prereqs → Steps → Result → Next Steps      | Walking through a task with numbered steps            | Gerund phrase: "Configuring [X]"     |
| Reference     | Parameter table or command syntax          | Providing lookup information (CLI, config, API)       | "[Command/Resource] Reference"       |
| Troubleshooting | Symptom → Cause → Resolution             | Diagnosing and fixing a known problem                 | Error message verbatim or symptom    |
| Release Notes | What's New → Changes → Known Issues        | Documenting changes per release                       | "Release [version] Notes"            |

### 7.4 Step Writing Rules (How-To Topics)

- Use **numbered lists** (`#.`) for ordered steps — never bullets for procedural content
- Each step = **one action**. Split compound actions into separate steps
- Begin each step with an **imperative verb**: "Run", "Edit", "Set", "Copy", "Navigate"
- Include the **expected result** after steps where verification is possible
- Use `.. code-block::` for all commands — never inline backtick-only code in steps

```rst
#. Install the required packages.

   .. code-block:: bash

      sudo dnf install -y omnia-control-plane

#. Verify the installation completed successfully.

   .. code-block:: bash

      omnia --version

   The output should display the installed version, for example: ``omnia 4.2.0``.
```

### 7.5 Code Blocks

Always specify the language:

```rst
.. code-block:: bash        ← Shell commands
.. code-block:: yaml        ← Configuration files
.. code-block:: json        ← JSON payloads
.. code-block:: python      ← Python scripts
.. code-block:: ini         ← INI/conf files
.. code-block:: text        ← Output / logs (no syntax highlighting)
.. code-block:: console     ← Interactive terminal sessions with prompts
```

**Rules:**
- Never use bare backtick blocks for multi-line commands
- Use `.. literalinclude::` to embed actual files from the repo (preferred for config examples)
- Use `.. code-block:: text` for sample output, never `bash`

### 7.6 Admonitions

Use sparingly. Each admonition must earn its place.

| Admonition       | Use For                                             | Frequency     |
|------------------|-----------------------------------------------------|---------------|
| `.. note::`      | Helpful context that is not critical                | ≤ 1 per page  |
| `.. warning::`   | Actions that could cause data loss or system issues | Only when true|
| `.. important::` | Prerequisites or dependencies that are easily missed| ≤ 1 per page  |
| `.. tip::`       | Optional shortcuts or best practices                | ≤ 1 per page  |
| `.. caution::`   | Potential for irreversible changes                  | Only when true|

**Never use `.. note::` as a substitute for main content.** If the information is important enough to note, it belongs in the body.

### 7.7 Cross-References

Use RST labels and `:doc:` or `:ref:` directives — never bare URLs to internal pages.

```rst
See :doc:`../reference/cli/omnia-provision` for CLI options.

For background, see :ref:`concept-cluster-topology`.
```

**Label naming convention:** `[topic-type]-[noun-phrase]`
- `concept-cluster-topology`
- `task-configure-network`
- `ref-omnia-cli`

### 7.8 Voice and Tone Rules

**General Principle:** Follow the Omnia Style Guide for general writing standards. The rules below are Omnia documentation-specific applications and clarifications.

**Style Guide Reference:** See `docs/.ai/STYLE_GUIDE.md` for comprehensive voice and tone guidance.

| Rule                          | Correct                                      | Incorrect                                   |
|-------------------------------|----------------------------------------------|---------------------------------------------|
| Use second person             | "Run the following command"                  | "The user should run..."                    |
| Use imperative mood for steps | "Configure the inventory file"               | "You can configure the inventory file"      |
| Use active voice              | "Omnia provisions the nodes"                 | "The nodes are provisioned by Omnia"        |
| Be direct and specific        | "Set `scheduler_type` to `slurm`"            | "Configure the scheduler appropriately"     |
| No marketing language         | "Omnia supports 200+ nodes per cluster"      | "Omnia's powerful engine scales effortlessly"|
| Neutral tone in troubleshooting | "This error occurs when..."               | "Unfortunately, this known issue..."        |
| Lead with the action          | "To configure X, edit the `omnia.yml` file"  | "The `omnia.yml` file can be used to configure X" |

### 7.9 ReadTheDocs / Sphinx-Specific Rules

- Use `conf.py` extensions: `sphinx.ext.intersphinx`, `sphinx_rtd_theme`, `myst_parser` (if Markdown mixed)
- Include `:orphan:` at the top of any `.rst` file intentionally excluded from a toctree
- Use `.. versionadded:: X.Y` and `.. versionchanged:: X.Y` for version-specific content
- Use `.. deprecated:: X.Y` for deprecated features, with a migration note
- All images must use `.. figure::` with an `:alt:` attribute and a caption

### Signal Sources and What They Drive

| Source Asset          | Location in Repo               | What It Informs                                  | Primary Doc Types Created        |
|-----------------------|-------------------------------|--------------------------------------------------|----------------------------------|
| BSpec (Technical Spec)| `docs/.ai/sources/bspec/`      | Technical accuracy, component behavior, data flow| Overview, Concepts, How-To, Reference, New Features |

| User Prompts          | Direct input                   | Specific documentation requests and updates      | All types                        |

**Note:** Omnia documentation is not triggered by GitHub signals. All documentation updates are initiated through direct prompts to WindSurf SWE.

### Source Priority When Inputs Conflict

1. **BSpec** — Technical ground truth. Always wins on implementation details.
2. **PRD** — Intent and scope. Wins on "what the feature should do" vs. internal details.
3. **Demo Transcripts** — User-facing language and workflows. Wins on how to describe behavior to end users.
4. **Existing Docs** — Style and placement precedent only.

### Content Type Decision Tree

```
New feature signal received?
│
├─ Does a BSpec exist? → YES → Use as primary source for technical accuracy
│                      → NO  → Flag for SME review; use PRD only
│
├─ Is there a Demo Transcript? → YES → Extract user-facing language and key workflows
│                              → NO  → Derive from PRD use cases
│
├─ Is this an update to existing content?
│   → YES → Load existing RST file, identify changed sections only
│   → NO  → Determine content type (§7.3) and placement (§6)
│
└─ Generate draft → Run CHECK against §11 checklists → Output with review markers
```

Since Omnia documentation is not triggered by GitHub signals, all documentation work is initiated through prompts. The AI should interpret prompts to determine:

| "Update release notes"                | Update release notes                    | Medium    | `overview/release-notes`    |

### 10.1 Good RST Examples

#### Concept Topic — Exemplary

```rst
.. _concept-cluster-provisioning:

Cluster Provisioning
====================

Cluster provisioning is the process by which Omnia configures bare-metal
nodes — installing operating systems, network settings, and required
software — to function as part of a managed cluster.

.. contents:: On This Page
   :local:
   :depth: 2

How Provisioning Works
-----------------------

When you run the Omnia provisioning playbook, Omnia performs the following
sequence:

1. Discovers nodes listed in the inventory file
2. Connects to each node via the baseboard management controller (BMC)
3. Installs the operating system using PXE boot
4. Applies the role-specific configuration (compute, storage, or login)
5. Registers the node in the cluster management database

Each step is idempotent — running the provisioning playbook again does
not re-provision nodes that are already correctly configured.

Prerequisites for Provisioning
--------------------------------

Provisioning requires:

* A correctly configured control plane (see :doc:`../getting-started/installation`)
* An inventory file with valid node IP addresses
* Network access from the control plane to all target nodes on the provisioning VLAN

Related Topics
--------------

* :doc:`../how-to/configuration/configure-inventory`
* :doc:`../reference/cli/omnia-provision`
```

**Why it's good:** Defines the concept clearly, explains the mechanism step-by-step, states prerequisites, uses approved terminology, and links to related topics.

#### How-To Topic — Exemplary

```rst
.. _task-configure-scheduler:

Configuring the Job Scheduler
==============================

Configure the job scheduler to enable workload management across the
cluster. Omnia supports Slurm and PBS Professional.

.. contents:: On This Page
   :local:
   :depth: 2

Prerequisites
-------------

Before configuring the scheduler:

* The control plane must be installed and running (see :doc:`../getting-started/installation`)
* All compute nodes must be provisioned (see :doc:`provision-nodes`)
* You must have root or sudo access on the control plane

Procedure
---------

#. Navigate to the Omnia configuration directory.

   .. code-block:: bash

      cd /opt/omnia/config

#. Open the scheduler configuration file.

   .. code-block:: bash

      sudo vi omnia.yml

#. Set the ``scheduler_type`` parameter to your preferred scheduler.

   .. code-block:: yaml

      scheduler_type: slurm

   Valid values are ``slurm`` and ``pbspro``.

#. Save the file and run the scheduler playbook.

   .. code-block:: bash

      ansible-playbook scheduler.yml -i inventory

#. Verify the scheduler is running.

   .. code-block:: bash

      sinfo

   The output lists all cluster partitions. If the command returns an
   error, see :doc:`../troubleshooting/scheduler-issues`.

Related Topics
--------------

* :doc:`../concepts/job-scheduling`
* :doc:`../reference/config/omnia-yml`
```

**Why it's good:** Clear prerequisites, one action per step, uses exact parameter names, includes verification, links to troubleshooting and reference.

```rst
Synopsis
--------

.. code-block:: text

   omnia provision [OPTIONS] [NODE...]

Options
-------

.. list-table::
   :header-rows: 1
   :widths: 25 15 60

   * - Option
     - Type
     - Description
   * - ``--inventory``
     - string
     - Path to the Ansible inventory file. Default: ``/opt/omnia/inventory``
   * - ``--dry-run``
     - flag
     - Simulate provisioning without making changes.
   * - ``--force``
     - flag
     - Re-provision nodes that are already configured.
   * - ``--log-level``
     - string
     - Verbosity level: ``info``, ``debug``, ``warn``. Default: ``info``

Examples
--------

Provision all nodes in the inventory:

.. code-block:: bash

   omnia provision --inventory /opt/omnia/inventory

Provision specific nodes:

.. code-block:: bash

   omnia provision node01 node02 node03

Simulate provisioning without changes:

.. code-block:: bash

   omnia provision --dry-run

Related Topics
--------------

* :doc:`../how-to/configuration/provision-nodes`
* :doc:`../concepts/cluster-provisioning`
```

**Why it's good:** Consistent structure, exact option names, default values documented, real examples.

### 10.2 Anti-Patterns to Avoid

| Anti-Pattern              | Bad Example                                                | Problem                                      | Correct Approach                                   |
|---------------------------|-----------------------------------------------------------|----------------------------------------------|----------------------------------------------------|
| Vague steps               | "Configure the network settings appropriately"            | Not actionable — user doesn't know what to do| "Set ``network_interface`` to ``ens3f0`` in `omnia.yml`"|
| Missing prerequisites     | (Steps with no prerequisites section)                     | User encounters unexpected failures          | List all required access, installed software, and prior steps |
| Marketing language        | "Omnia's powerful, industry-leading provisioning engine"  | Not factual, not useful                      | "Omnia provisions nodes using Ansible playbooks"   |
| Passive voice             | "The scheduler should be configured before nodes are added"| Unclear who acts and when                  | "Configure the scheduler before adding nodes"      |
| Bare URLs                 | `See https://docs.omnia.io/reference/cli`                 | Breaks on restructure; not semantic          | `See :doc:`../reference/cli/omnia-cli``            |
| Undeclared acronyms       | "Use the BMC to configure IPMI over the OOB VLAN"        | Inaccessible to new users                    | Spell out on first use per document                |
| Generic code blocks       | `` ` `` without language                                  | No syntax highlighting; unclear context      | Always use `.. code-block:: bash` (or appropriate language) |
| Step combining            | "Open the file and change the scheduler setting and save" | Skips verification opportunities             | One action per numbered step                       |

## 11. Guardrails and Constraints

### 11.1 Always Do

- [ ] Include a `Prerequisites` section in every How-To topic
- [ ] Specify the language in every `.. code-block::` directive
- [ ] Add the new file to the parent `index.rst` toctree
- [ ] Add a unique RST label to the top of every file: `.. _[unique-label]:`
- [ ] Use only approved terminology from §9
- [ ] Include a `Related Topics` section at the bottom of every page
- [ ] Use exact configuration parameter names with double backticks
- [ ] Add `.. versionadded::` or `.. versionchanged::` when content is version-specific
- [ ] Validate placement against the Information Map (§6) before creating a new file
- [ ] Write from source assets (HLD → Engineering Notes → Demo Transcriptions → Unit Tests) — never from general knowledge alone

### 11.2 Never Do

- [ ] Use bare URLs to link between internal documentation pages
- [ ] Skip the prerequisites section on How-To pages
- [ ] Reference internal team names, personal names, or internal system aliases
- [ ] Include internal repo paths not intended for end users
- [ ] Use marketing superlatives: "powerful", "seamless", "industry-leading", "effortlessly"
- [ ] Combine multiple actions into a single numbered step
- [ ] Create an `.rst` file without registering it in a toctree
- [ ] Use `.. note::` as a container for main content
- [ ] Invent parameter names — always verify against BSpec or actual config files
- [ ] Write troubleshooting content that blames the user

### 11.3 Confidence Thresholds

| Confidence Level | Threshold | AI Action                                                          |
|------------------|-----------|--------------------------------------------------------------------|
| High             | > 0.85    | Generate normally                                                  |
| Medium           | 0.6–0.85  | Generate with `.. AI_REVIEW::` comment markers                    |
| Low              | < 0.6     | Flag entire section for human authoring; provide outline only      |

### 11.4 Review Markers

When uncertain, insert an RST comment:

```rst
.. AI_REVIEW: [reason for uncertainty]
```

**Common reasons:**
- `Inferred from Demo Transcript only — verify against BSpec`
- `No BSpec found for this component — SME verification required`
- `Parameter name not confirmed — check omnia.yml schema`
- `Multiple possible workflows — confirm correct path with SME`
- `Placement uncertain — verify toctree location`
- `Version applicability unclear — confirm which Omnia version`
